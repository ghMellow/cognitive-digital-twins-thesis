Prima del 2017: Architettura RNN

  • Processamento del linguaggio: parola per parola attraverso reti ricorrenti
  • Principio: mantenere stato interno che accumula il contesto
  • Limite strutturale: lo stato si comprime, i gradienti svaniscono
  • Dipendenze a lunga distanza perse — soggetti e verbi separati da molte parole
  • Esempio: "Il server che gestisce il cluster... è caduto" — già un problema con 12 parole tra soggetto e verbo

Il Transformer: L'Attenzione è Tutto

  • Idea centrale: ogni token osserva tutti gli altri contemporaneamente
  • Ogni token produce: vettori Query, Key, Value
  • La compatibilità Query-Key determina il peso dell'attenzione distribuita sui Values
  • Parallelizzabile → scalabile a GPU → dataset massicci
  • Risultato: rappresentazione contestuale — stessa parola ha embedding diversi in contesti diversi
  • "Banco di pesci" vs. "conto bancario" — l'LLM comprende le sfumature

La Conoscenza Nascosta negli LLM

  • Obiettivo di training: predire il token successivo (apparentemente semplice)
  • Risultato emergente: semantica, ragionamento analogico, strutture logiche, conoscenza enciclopedica
  • Insight chiave: predire correttamente richiede comprensione profonda del contesto
  • Conseguenza fondamentale: la semantica è implicita nel modello, non in strutture formali
  • Nessun knowledge graph, nessuna ontologia, nessuna regola dichiarata
  • Concetti semanticamente simili occupano regioni vicine nello spazio vettoriale

Tre Confini Architetturali

  • Nessuna memoria persistente — ogni sessione riparte da zero
  • Nessuna capacità di agire nel mondo — non può leggere file, chiamare API, modificare sistemi
  • Finestra di contesto finita — 100K-200K token tipici (fino a 1M+ in modelli specializzati)
  • Categoria emergente: Reasoning models (o1, DeepSeek R1, Gemini 2.5 Thinking)
  • Questi non sono bug. Sono i confini del componente. Comprenderli è essenziale.

L'Agente: LLM come Cervello in un Loop

  • Definizione precisa: Un while loop con un LLM come cervello
  • Nome del pattern: ReAct (Reason + Act)
  • Ricevi obiettivo → LLM decide azione → Esegui → Osserva risultato → Reinietta contesto → Loop finché obiettivo raggiunto
  • Differenza da singola chiamata LLM: persistenza, azione, correzione basata su osservazione
  • Loop continua autonomamente; nessuna logica hardcoded

Un Agente in Azione

Obiettivo: Leggi config.json, cambia debug da false a true, salva.

  • Iterazione 1: LLM legge obiettivo → "Devo il contenuto del file" → invoca read_file()
  • Iterazione 2: LLM riceve contenuto → "Vedo che debug è false" → invoca write_file()
  • Iterazione 3: LLM verifica → invoca read_file() di nuovo → obiettivo raggiunto
  • Risultato: Tre decisioni autonome basate su osservazioni, nessun step hardcoded

Strumenti: Function Calling e MCP

  • Un agente non può operare senza strumenti — le funzioni che può invocare
  • Function calling: LLM genera chiamate di funzione strutturate, il programma le intercetta ed esegue
  • Prima del 2024: ogni framework definiva il proprio protocollo
  • Novembre 2024: Anthropic lancia Model Context Protocol (MCP) — standard aperto
  • 2025: OpenAI, Google DeepMind lo adottano → standard de facto
  • Analogia: Come USB-C — un'unica interfaccia per tutto

Skills: Prompting Strutturato e Ripetibile

  • Definizione: procedure condensate in Markdown per operazioni ricorrenti
  • Non codice, solo istruzioni di prompting: come affrontare problemi
  • Esempi:
    • Ingest paper → identifica concetti → aggiorna wiki → registra nel log
    • Warm-up contesto → carica CLAUDE.md → leggi scaffolding → riassumi stato
  • Valore: Chiunque può leggere, adattare e condividere versioni migliori
  • Utilità per pattern generalizzabili e ripetibili

Memoria: Il Problema Aperto

  • LLM non ha stato tra sessioni — ogni conversazione riparte da zero
  • Per chatbot generici: accettabile
  • Per knowledge base: problema critico
  • Tre dimensioni:
    • Cosa memorizzare — la selezione non è banale
    • Come organizzarlo — la struttura influenza la qualità del ragionamento
    • Come recuperarlo — la finestra di contesto è finita

Tre Categorie di Memoria

  • Contestuale — solo nella sessione corrente, buffer temporaneo (conversazione)
  • Statica — permanente, raramente cambia (system prompt, regole, procedure)
  • Episodica — permanente, mutabile, accumula nel tempo (log, decisioni, fatti)

L'arte del design agente: decidere cosa va dove, e come recuperarlo quando necessario.

RAG vs. Approccio LLM Wiki

  • RAG: indicizza documenti in vector store → recupera chunk simili → inietta in contesto
  • Limite: recupera frammenti, non sintesi; qualità dipende dall'embedding
  • LLM Wiki: costruisci e mantieni wiki sintetica come struttura Markdown
  • Pattern: raw/ (originali immutabili) + wiki/ (elaborazione sintetizzata)
  • Formalizzazione recente: Andrej Karpathy (aprile 2026) — separazione a due livelli
  • Risultato: memoria che compila, non recupera

Posizionamento Tre Approcci di Memoria

  • Markdown/Wiki: decine-centinaia di doc, gerarchico, zero infrastruttura
  • RAG: migliaia-milioni di doc, retrieval semantico flat, vector store esterno
  • Palazzo della Memoria: centinaia con riferimenti densi, organizzazione spaziale, SQLite + ChromaDB locale
  • Palazzo della Memoria: frontiera emergente — combina struttura + semantica + compressione

Il Mercato: Packaging del Blob

  • 2025-2026: i framework per agenti non sono ancora standardizzati — un "blob" di pattern
  • Protagonisti: LangChain, LangGraph, CrewAI, Claude Code, smolagents
  • Verità chiave: tutti i framework si riducono alla stessa struttura
  • Perché i framework esistono:
    • Commerciale: la potenza ha bisogno di packaging → documentazione → supporto
    • Usabilità: il software ha bisogno di interfacce stabili (DB ha SQL, OS ha shell)

Mettendo Tutto Insieme: Workflow Pratico

  • raw/ — documenti sorgente immutabili (paper, call, trascrizioni)
  • wiki/ — elaborazione sintetizzata (fonti, concetti, analisi + log)
  • CLAUDE.md — manuale operativo (ruolo, contesto, procedure)
  • Skills: procedure prompting ripetibili in Markdown
  • Bootstrap agente: carica CLAUDE.md → leggi wiki/index → leggi log recente → capisci stato → agisci

Punti Chiave

  • Transformer: il meccanismo di attenzione risolve le dipendenze a lunga distanza
  • LLM: imparano la semantica implicitamente; nessuna memoria persistente o capacità di azione
  • Agenti: while loop con cervello LLM — pattern ReAct
  • Strumenti: function calling standardizzato via MCP (2024-2025)
  • Memoria: tre approcci — Markdown Wiki il migliore per workflow controllati e personali
  • Framework: tutti si riducono alla stessa struttura: LLM + Memoria + Strumenti
  • Il nocciolo: comprendi prima le fondamenta, poi i framework
